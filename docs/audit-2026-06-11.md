### C1 — Gemini API key leaked in URL query parameter

**File**: `src/embedder/gemini.rs:81-87, 90-97`
**Severity**: CRITICAL (security — credential leak)

**What's wrong**:
The Gemini API key is embedded in the request URL as a `?key=` query parameter.
This leaks the key into:
- `reqwest` debug logs (full URL visibility)
- Error messages from transport failures (reqwest includes the URL in `Display`)
- Intermediate proxy/CDN access logs
- Process inspection (`/proc/<pid>/cmdline`, core dumps)
- `tracing` spans at debug level

OpenAI (`openai.rs:118`) uses `.bearer_auth()` (HTTP `Authorization` header) — correct and secure.

Google's Gemini API docs recommend `x-goog-api-key` header or `Authorization: Bearer` for OpenAI-compatible endpoints. The `?key=` query parameter is explicitly documented as "not recommended for production."

**Fix**:

In `src/embedder/gemini.rs`:

1. Remove the `?key={key}` from `embed_url()` and `batch_url()`:
   ```rust
   fn embed_url(&self) -> String {
       format!("{}/models/{}:embedContent", Self::BASE_URL, self.model)
   }

   fn batch_url(&self) -> String {
       format!("{}/models/{}:batchEmbedContents", Self::BASE_URL, self.model)
   }
   ```

2. Add the `x-goog-api-key` header in both `embed()` and `embed_batch()`:
   ```rust
   let response = self
       .client
       .post(&url)
       .header("x-goog-api-key", &self.api_key)
       .json(&body)
       .send()
       .await
       // ...
   ```

3. Update tests in `gemini_embed_url_contains_model_and_key()` and `gemini_batch_url_contains_batch_endpoint()`:
   - Remove `url.contains("my-api-key")` assertion — key is no longer in URL
   - Add assertion that URL does NOT contain the key

**Verification**:
```bash
cargo test gemini
cargo test --lib  # all 461 tests
```

**Reference**: https://ai.google.dev/gemini-api/docs/api-key (recommends `x-goog-api-key` header)

### C2 — Missing HTTP timeouts on all reqwest clients

**Files**: `src/embedder/gemini.rs:31`, `src/embedder/openai.rs:26`, `src/embedder/ollama.rs` (client field), `src/embedder/model_manager.rs:103`
**Severity**: CRITICAL (production outage — hung connection blocks all retries)

**What's wrong**:
Every `reqwest::Client::new()` in the embedders and model manager creates a client with **no timeout**. A hung TLS handshake, slow-loris server, or network partition causes:
1. One `embed()` call blocks indefinitely
2. The retry loop retries 5+ times, each attempt also unbounded
3. The calling tokio task (indexer) is wedged forever
4. For ONNX: `Mutex<Session>` serializes all calls, so one hung task blocks the entire pipeline

**Fix**:

Add a shared timeout configuration. Create a helper in `src/embedder/http.rs`:

```rust
use std::time::Duration;

/// Default HTTP timeout for embedding provider requests.
pub const REQUEST_TIMEOUT: Duration = Duration::from_secs(60);

/// Default connect timeout.
pub const CONNECT_TIMEOUT: Duration = Duration::from_secs(10);

/// Build a reqwest::Client with sensible defaults for embedding providers.
pub fn build_http_client() -> reqwest::Client {
    reqwest::Client::builder()
        .timeout(REQUEST_TIMEOUT)
        .connect_timeout(CONNECT_TIMEOUT)
        .build()
        .expect("reqwest::Client::builder with valid config should not fail")
}
```

Then replace every `reqwest::Client::new()` with `build_http_client()`:
- `gemini.rs:49` — `GeminiEmbedder::new()`
- `openai.rs:44` — `OpenAiEmbedder::new()`
- `ollama.rs` — `OllamaEmbedder::new()`
- `model_manager.rs:103` — `fetch_with_progress()`: the model download path has its own timeout requirement. Use `reqwest::Client::builder().connect_timeout(Duration::from_secs(10)).timeout(Duration::from_secs(300)).build().expect(...)` since model downloads can be large.

**Verification**:
```bash
cargo build --release
cargo test --lib
```

### C4 — JSON-RPC 2.0 violation: notifications receive error responses

**File**: `src/mcp/mod.rs:133-137`
**Severity**: CRITICAL (protocol violation — breaks MCP clients)

**What's wrong**:
```rust
other => {
    let response = make_error_result(
        request.id.clone(),
        -32601,
        format!("Method not found: {method}"),
    );
    write_message(&stdout_handle, &response).await?;
}
```
Any notification method other than `notifications/initialized` (e.g., `notifications/cancelled`, `notifications/progress`) falls into the `other` arm and the server sends an error response. JSON-RPC 2.0 §4.1: notifications MUST NOT receive a response.

**Fix**:

Detect notifications by checking both method prefix AND the `id` field:

```rust
other => {
    let is_notification = method.starts_with("notifications/");
    if is_notification {
        // Notifications must not receive a response (JSON-RPC 2.0 §4.1)
        tracing::debug!("Received notification: {method}");
        continue;  // or return Ok(()) depending on control flow
    }
    let response = make_error_result(
        request.id.clone(),
        -32601,
        format!("Method not found: {method}"),
    );
    write_message(&stdout_handle, &response).await?;
}
```

Note: This also fixes the `notifications/initialized` arm at lines 126-132, which currently sends a `{}` response (which the comment says it shouldn't, but the code does).

**Verification**:
- Send a `{"jsonrpc":"2.0","method":"notifications/cancelled"}` via stdin and confirm no response is written
- `cargo test mcp`

### C5 — Silent request loss: `id: null` misclassified as notification

**File**: `src/mcp/mod.rs:97-100, src/mcp/schema.rs` (JsonRpcRequest struct)
**Severity**: CRITICAL (data loss — client never gets response)

**What's wrong**:
```rust
let is_notification = request.id.is_null();
```
`JsonRpcRequest.id` is `serde_json::Value` with `#[serde(default)]`. Both a missing `id` field AND an explicit `{"id": null}` deserialize to `Value::Null`. `is_null()` returns true for both.

Per JSON-RPC 2.0 §4.1: a request without an `id` is a notification (no response). A request WITH an `id` (even `null`) MUST receive a response. The current code treats `{"id": null, "method": "tools/list"}` as a notification and never responds.

**Fix**:

Change `JsonRpcRequest.id` from `serde_json::Value` (with `#[serde(default)]`) to `Option<serde_json::Value>`:

In `src/mcp/schema.rs`:
```rust
pub struct JsonRpcRequest {
    pub jsonrpc: String,
    #[serde(default)]
    pub method: String,
    pub id: Option<serde_json::Value>,  // None = notification, Some(_) = request
    pub params: serde_json::Value,
}
```

In `src/mcp/mod.rs`, change detection:
```rust
let is_notification = request.id.is_none();
```

And update the error response construction to use `request.id.clone()` (which is now `Option<Value>`).

**Verification**:
```bash
cargo test mcp
```

### C7 — Watcher `pending_files` is dead code — tokio runtime unreachable

**File**: `src/watcher/mod.rs:107-120`
**Severity**: CRITICAL (hidden feature completely non-functional)

**What's wrong**:
```rust
// Spawn a blocking task to update pending (since we're in a sync callback)
std::thread::spawn({
    let pending_clone = pending_clone.clone();
    move || {
        let rt = tokio::runtime::Handle::try_current();
        if let Ok(handle) = rt {
            handle.block_on(async {
                pending_clone.write().await.extend(new_paths.iter().cloned());
            });
        }
    }
});
```
The notify-debouncer callback runs on a **dedicated std::thread** (not a tokio worker). `Handle::try_current()` returns `Err` because there's no tokio runtime on that thread. The `if let Ok(handle)` branch is **never taken**.

Result: `pending_files()` always returns an empty list. The staleness banner in MCP responses never shows pending files. The watcher background loop never reconciles them.

**Fix**:

Use a `tokio::sync::mpsc` channel instead of trying to access the runtime from a foreign thread:

```rust
// In FileWatcher struct, add a second channel:
// tx_pending: tokio::sync::mpsc::UnboundedSender<Vec<PathBuf>>

// In the notify callback:
let new_paths: Vec<PathBuf> = events.iter()
    .filter_map(|e| {
        let p = e.path.strip_prefix(&root_clone).ok()?;
        if p.is_file() { Some(p.to_path_buf()) } else { None }
    })
    .collect();

if !new_paths.is_empty() {
    let _ = tx_pending.send(new_paths.clone());
}

// In the watcher background loop, receive from both channels:
tokio::select! {
    Some(batch) = rx.recv() => { /* reindex batch */ }
    Some(pending) = rx_pending.recv() => { 
        self.pending.write().await.extend(pending);
    }
}
```

This removes the `std::thread::spawn` entirely and keeps all watcher logic on tokio.

**Verification**:
- Write a test that modifies a file, waits for the watcher event, then checks `pending_files().len() > 0`
- Since the current tests don't verify this (they check empty pending which is always true), add a dedicated test

### H8 — Gemini/OpenAI retry loop wastes last attempt

**Files**: `src/embedder/gemini.rs:190-200`, `src/embedder/openai.rs:145-156`
**Severity**: HIGH (logic error — wasted retry, misleading error)

**What's wrong**:
When `attempt == MAX_RETRIES` and `should_retry(status)` is true, the condition `attempt < MAX_RETRIES` is false, so it falls through to the error branch — even though the API just returned a retryable response. The last retry attempt is wasted.

The loop runs `0..=MAX_RETRIES` (6 iterations for `MAX_RETRIES=5`). On the 6th attempt, the response is discarded and reported as error. The error message says "API error (HTTP 503)" when it should say "max retries exceeded".

**Fix**:
Change the condition to check first, then sleep if there are more retries left:
```rust
if should_retry(status) {
    if attempt < MAX_RETRIES {
        let backoff = calculate_backoff(attempt, BASE_BACKOFF_MS, MAX_BACKOFF_MS, jitter_factor());
        tokio::time::sleep(backoff).await;
        continue;
    }
    return Err(VectorCodeError::EmbedderError {
        message: format!("{provider} API: max retries ({MAX_RETRIES}) exceeded (HTTP {status})"),
    });
}
```

**Verification**: Test with a mock HTTP server returning 503 on every request.

### H9 — No `Retry-After` header respect — `RateLimited` variant is dead code

**Files**: `src/embedder/gemini.rs:208-260`, `src/embedder/openai.rs:163-216`, `src/error.rs:25-26`
**Severity**: HIGH (rate-limit handling broken)

**What's wrong**:
On HTTP 429, the retry loop uses fixed exponential backoff (`1s, 2s, 4s, 8s, ...`), ignoring the server's `Retry-After` header. If the server says "wait 60 seconds", the client retries after ~1 second, spams the endpoint, burns all retry attempts in under 10 seconds, and fails hard.

`VectorCodeError::RateLimited { retry_after_secs: u64 }` is defined in `src/error.rs` but **never constructed anywhere**.

**Fix**:

1. After 429 response, read `Retry-After` header:
```rust
if status == 429 {
    let retry_after = response
        .headers()
        .get("retry-after")
        .and_then(|v| v.to_str().ok())
        .and_then(|v| v.parse::<u64>().ok());
    
    if let Some(seconds) = retry_after {
        if seconds <= 120 && attempt < MAX_RETRIES {
            tokio::time::sleep(Duration::from_secs(seconds)).await;
            continue;
        }
    }
}
```

2. When no `Retry-After` or value > 120s, fall back to exponential backoff, but at minimum use `MAX_BACKOFF_MS` (60s) for 429s:
```rust
let backoff = calculate_backoff(attempt, BASE_BACKOFF_MS, MAX_BACKOFF_MS, jitter_factor());
let effective = if status == 429 { backoff.max(Duration::from_secs(30)) } else { backoff };
```

3. Return `VectorCodeError::RateLimited` when rate limit is hit after max retries.

**Verification**: Test with mock HTTP server returning 429 with `Retry-After: 5`.

**What's wrong**:
```rust
if let Err(e) = tx.try_send(new_paths) {
    tracing::warn!("Watcher channel full, dropping {} paths: {e}", new_paths.len());
}
```
Channel capacity is 64 batches. A `git checkout` of a large repo can produce dozens of batches rapidly. When the buffer is full, batches are silently dropped. No retry, no backpressure, no overflow-to-disk.

**Fix**:
- Increase channel capacity to 256 or 512
- On `TrySendError::Full`, log WARN but also store the paths in a secondary `VecDeque` and retry on the next iteration of the watcher loop
- Or use `tokio::sync::mpsc::channel` with a larger buffer and `.send().await` (blocks notify thread briefly but safe for low-latency filesystem events)

**Verification**: Run `git checkout` on a repo with 1000+ files, verify watcher processes all batches without dropping.

### H16 — TOCTOU race in `vectorcode init`

**File**: `src/cli/init.rs:46-94`
**Severity**: HIGH (data corruption on concurrent init)

**What's wrong**:
```rust
if vc_dir.exists() { return Err(...) }
create_dir_all(&vc_dir)?;
Database::open(&db_path)?;
std::fs::write(&config_path, ...)?;
```
Two concurrent `init` invocations both pass `exists()`, both create the directory (idempotent), both create the database, and the second `write` clobbers the first's `config.toml`.

**Fix**:
Use `OpenOptions::create_new(true)` (O_EXCL) on a lock file:
```rust
let lock_path = vc_dir.join(".init.lock");
let _lock = std::fs::OpenOptions::new()
    .create_new(true)
    .write(true)
    .open(&lock_path)
    .map_err(|_| VectorCodeError::Config(
        "Another init is running or .vectorcode/ already exists. Remove it and retry.".into()
    ))?;
// ... proceed with init ...
// Lock file is automatically cleaned up when `_lock` drops
```

**Verification**: Run two `vectorcode init` concurrently, confirm one fails with a clear error.

### H17 — Self-update has no signature/checksum verification

**File**: `src/cli/upgrade.rs:111-224`
**Severity**: HIGH (supply-chain security)

**What's wrong**:
The tarball is fetched over HTTPS and its bytes are trusted unconditionally. No SHA-256, minisign, or GPG verification. A compromised release pipeline or MITM with leaked cert yields arbitrary code execution.

The homebrew formula (`homebrew/vectorcode.rb:9,12`) has `PLACEHOLDER_ARM64_SHA256` — the project knows it should verify but doesn't.

**Fix**:

1. Publish a `SHA256SUMS` file alongside each release on GitHub
2. Download and verify before extracting:
```rust
// Fetch SHA256SUMS
let sums_url = format!("{base}/SHA256SUMS", ...);
let sums = reqwest::get(&sums_url).await?.text().await?;

// Find the expected hash for our binary
let expected = parse_sha256_for_target(&sums, target)?;

// Compute hash of downloaded bytes
let actual = blake3::hash(&bytes).to_hex(); // or sha2::Sha256

if actual != expected {
    return Err("Checksum mismatch: possible tampering".into());
}
```

**Verification**: Manually test with a known-good release.

### H18 — Upgrade no-op on macOS

**File**: `src/cli/upgrade.rs:69`
**Severity**: HIGH (silent failure)

**What's wrong**:
```rust
self_replace::self_replace(&binary_path)?;
```
On macOS, `self_replace` uses `rename(2)`. The running process keeps the old inode memory-mapped. The on-disk binary is updated, but the current process (and any process that exec's from the same shell) still runs the old version. Subsequent `vectorcode upgrade` calls from the same shell launch the old binary.

**Fix**:
After `self_replace`:
1. Log a strong recommendation: "Upgrade complete. Please restart your terminal or run `exec $SHELL -l` to use the new version."
2. For the current process: `std::process::Command::new(std::env::current_exe()?).spawn()` — but this is tricky because the old process is still running. Better to just document and rely on the user restarting.

**Verification**: Run upgrade on macOS, verify the message is shown.

### H19 — `concurrency` config field is dead code

**File**: `src/engine/indexer.rs:202-308`, `src/config/schema.rs` (IndexingConfig), `src/cli/index.rs`
**Severity**: HIGH (misleading config — user expects parallelism, gets none)

**What's wrong**:
`IndexingConfig.concurrency` (default 8) is exposed in config.toml, CLI flags (`--concurrency`), and schema docs. But `process_file_entries` is a strictly sequential `for` loop — no `tokio::spawn`, no `buffer_unordered`, no `join_all`. Setting `concurrency: 16` provides zero speedup.

**Fix**:
Either:
- **Implement parallelism**: use `futures::stream::iter(files).map(|f| process_file(f)).buffer_unordered(concurrency)`
  - Note: this requires careful SQLite connection handling (one conn per task or a connection pool)
- **Remove the field** and document single-threaded indexing until parallelism is implemented

**Verification**: If implementing, benchmark before/after with a large project.
